# Module Contract — Evidence Graph

**Status:** Draft — pending review before implementation
**Milestone:** 3, Module 4 of 10 (see Milestone_2_Architecture.md Section 11)

**Note on module nature:** like Conversation Memory, this is a
deterministic storage/query module, not an LLM extractor. No
`ProviderAdapter` usage. **Identity discipline (per explicit review
guidance before this contract was written):** evidence entries
reference stable IDs from upstream modules (`turn_id`, `question_id`,
`competency_id`) rather than copying or denormalizing conversation
content. The one deliberate exception is `evidence_excerpt` — see
below for why a short quoted excerpt is necessary despite this rule,
and how it's kept honest.

---

## Purpose

Structured record linking every piece of extracted evidence to its
source — the mechanism that makes Evidence-Based Feedback (Architecture
Review Gate #4) structural rather than reconstructed after the fact.
Per Milestone 2 Architecture Section 2.4: "every feedback statement is
generated by querying this graph, not by asking the model to
'summarize how it went.'"

This module does not decide what counts as evidence (Evaluation
Engine's job, Module 6 — not yet built) or what confidence a piece of
evidence produces (Competency Model's job, Module 7). This module only
stores, links, and queries evidence entries that some other module
(eventually Evaluation Engine) writes to it. **This module is built
before its primary writer exists** — same relationship Conversation
Memory has to the modules that will call it — so its write API must be
usable by a future module without requiring changes here.

## Inputs

### `add_evidence(...)`
| Field | Type | Required | Notes |
|---|---|---|---|
| interview_id | string | Yes | |
| competency_id | string | Yes | References a competency from JD Understanding's output — this module does not validate that the ID actually exists in a given interview's competency set; that's the caller's responsibility |
| turn_id | string | Yes | References a `TurnRecord` from Conversation Memory — the interaction this evidence came from |
| question_id | string | Yes | Denormalized alongside turn_id deliberately — see Identity Discipline note below |
| evidence_excerpt | string | Yes | A short, directly-quoted substring from the answer this evidence is drawn from — see Identity Discipline note |
| relation | enum | Yes | `supports` \| `contradicts` — per Architecture Section 2.4's conflict-flagging requirement |
| contradicts_evidence_id | string | No | Required if `relation == "contradicts"` — points to the specific prior evidence entry this one conflicts with, not just a general "something's off" flag |

### `get_evidence_for_competency(interview_id, competency_id)`
Returns all evidence entries for a competency, ordered by the sequence
of the turns they came from.

### `get_evidence_for_turn(interview_id, turn_id)`
Returns all evidence entries extracted from a specific turn.

### `has_contradictions(interview_id, competency_id)`
Returns whether any `contradicts` relation exists for a competency —
this is the specific query the Reasoning Engine uses to decide whether
"challenge inconsistency" is an available strategy (Architecture
Section 2.5).

## Output — Evidence Entry Schema

```json
{
  "evidence_id": string,
  "interview_id": string,
  "competency_id": string,
  "turn_id": string,
  "question_id": string,
  "evidence_excerpt": string,
  "relation": "supports" | "contradicts",
  "contradicts_evidence_id": string | null,
  "created_at": string
}
```

### Identity Discipline — what's referenced vs. what's duplicated

Per explicit guidance: this module references stable IDs rather than
copying conversation content. Concretely:
- **Referenced, not copied:** the full question text, the full answer
  text, the competency's name/emphasis, anything already owned by
  Conversation Memory or JD Understanding. If a caller (eventually
  Feedback Generator) needs the full question or answer, it looks that
  up via `turn_id` against Conversation Memory — this module does not
  keep its own copy.
- **The one deliberate exception: `evidence_excerpt`.** A short,
  directly-quoted substring of the answer is stored here, not just a
  reference. This is necessary because "evidence" is not the same
  thing as "the whole answer" — a single answer might produce zero,
  one, or several evidence entries, each pointing at a different part
  of what was said, sometimes for different competencies. Storing only
  `turn_id` would lose which *part* of the answer is the actual
  evidence. This is not a violation of the identity-discipline
  guidance — it's evidence *content*, not duplicated conversation
  metadata.
- **Enforced honesty on the exception:** `evidence_excerpt` must be
  independently verifiable against the real answer text stored in
  Conversation Memory — reusing the shared provenance validator
  (`app.engine.shared.validator`, already proven in Resume and JD
  Understanding) to confirm `evidence_excerpt` is an actual substring
  of the turn's `answer_text`, not a fabricated or paraphrased
  summary. This is the same no-fabrication discipline applied to a new
  context.
- **`question_id` alongside `turn_id`:** technically `question_id` is
  derivable by looking up `turn_id` in Conversation Memory, so storing
  both is a mild, deliberate denormalization — justified because
  `get_evidence_for_turn` and competency-scoped queries are expected to
  be frequent, and requiring a Conversation Memory lookup on every
  Evidence Graph read would create a hard runtime dependency between
  two modules that should otherwise be independently queryable. This
  tradeoff is stated explicitly rather than left implicit.

## Error Contract

| Code | Meaning | Behavior |
|---|---|---|
| `TURN_NOT_FOUND` | `add_evidence()` called with a `turn_id` that doesn't correspond to any known turn | Reject — raise. Evidence cannot exist without a real source turn |
| `EXCERPT_NOT_TRACEABLE` | `evidence_excerpt` fails the provenance check against the turn's actual `answer_text` | Reject — raise. This is the enforcement mechanism for the no-fabrication rule in this module, same severity as Resume/JD Understanding's equivalent check |
| `CONTRADICTS_TARGET_NOT_FOUND` | `relation == "contradicts"` but `contradicts_evidence_id` doesn't reference an existing evidence entry | Reject — raise |
| `MISSING_CONTRADICTION_TARGET` | `relation == "contradicts"` but `contradicts_evidence_id` is null | Reject — raise. A contradiction claim without a specific target is not a valid conflict flag per the architecture's requirement that this be a "reference flag," not a vague signal |

All four are rejection errors — unlike Resume/JD Understanding, this
module has no "recoverable" error tier, because evidence entries with
unverifiable claims are not a degraded-but-useful state; they're
exactly the kind of fabricated data the whole no-fabrication discipline
exists to prevent from entering the system.

## Responsibilities

- Store evidence entries with correct references to their source turn
  and target competency.
- Enforce that every `evidence_excerpt` is traceable to real answer
  text, reusing the shared validator.
- Track `supports`/`contradicts` relations, including which specific
  prior evidence a contradiction targets.
- Provide competency-scoped, turn-scoped, and contradiction-check
  queries.
- Provide a simple, queryable data structure — per the architecture,
  "not a full contradiction-resolution algorithm."

## Explicit Non-Responsibilities

- Does not decide what counts as evidence or extract it from answer
  text (Evaluation Engine's job — this module only stores what it's
  given).
- Does not compute competency confidence scores (Competency Model's
  job).
- Does not decide whether a contradiction is significant enough to
  probe (Reasoning Engine's job — this module only exposes
  `has_contradictions()` as a fact, not a judgment about what to do
  with it).
- Does not perform semantic contradiction detection itself — if two
  evidence entries conflict, the caller (eventually Evaluation Engine)
  determines that and tells this module via `relation: "contradicts"`
  plus a specific `contradicts_evidence_id`. This module does not
  infer contradictions on its own.
- Does not generate the reasoning trace/log entry (Logging module's
  job, Module 5).
- Does not validate that a given `competency_id` actually belongs to
  the interview's JD-derived competency set — that's a cross-module
  integrity concern left to the caller for v0.1, not enforced here.

## Acceptance Criteria

1. `add_evidence()` with a valid `turn_id` and a traceable
   `evidence_excerpt` succeeds and is retrievable via both
   `get_evidence_for_competency` and `get_evidence_for_turn`.
2. `add_evidence()` with an `evidence_excerpt` that is NOT a substring
   of the referenced turn's actual `answer_text` raises
   `EXCERPT_NOT_TRACEABLE` — tested with a fabricated excerpt against
   a real Conversation Memory turn, reusing the shared validator's
   `is_traceable()` logic directly (not reimplemented).
3. `add_evidence()` with a nonexistent `turn_id` raises `TURN_NOT_FOUND`.
4. `relation: "contradicts"` without `contradicts_evidence_id` raises
   `MISSING_CONTRADICTION_TARGET`.
5. `relation: "contradicts"` with a `contradicts_evidence_id` that
   doesn't exist raises `CONTRADICTS_TARGET_NOT_FOUND`.
6. `has_contradictions()` correctly returns `True` only when a real
   `contradicts` relation exists for that competency, `False`
   otherwise — both directions tested.
7. `get_evidence_for_competency` results are ordered by the sequence
   of their source turns (reusing Conversation Memory's
   `sequence_number` for ordering, not a separately-tracked order).
8. Evidence entries do not duplicate full question/answer text — a
   structural test confirms the schema contains no field holding a
   complete question or answer, only `evidence_excerpt` (a short
   fragment) and ID references. This is the direct test of the
   Identity Discipline principle stated above, not just a design intent.
9. Two different `interview_id`s never see each other's evidence —
   same isolation discipline as Conversation Memory's AC10, same
   severity (real correctness bug if missed, not minor).
