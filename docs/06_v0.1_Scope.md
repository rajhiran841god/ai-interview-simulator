# v0.1 Scope — Engine-First Pilot

**Status:** Locked pending Stage 1 pilot results (see Decision #002)
**Supersedes:** earlier MVP lists that included avatar/voice in v0.1

---

## What v0.1 Actually Is

A **pilot usability study**, not a full product launch. Goal: validate
whether the Interview Intelligence Engine — not any particular
interface — produces a genuine "this felt like a real interview"
reaction in real MBA students.

## In Scope

1. Sign up / sign in
2. Resume upload
3. Job description upload
4. Start interview (text interface only)
5. Engine asks adaptive, resume/JD-grounded questions
6. Engine performs cross-questioning / intelligent follow-ups based on
   the candidate's actual answers (not scripted)
7. Engine ends interview
8. Feedback report generated
9. Interview transcript stored
10. Basic interview history (view past attempts)

## Explicitly Deferred (not cut — deferred, pending Stage 1 evidence)

- Voice interface
- Avatar / video interviewer
- AR/VR
- Recruiter dashboard
- Placement officer / TPO dashboard
- Payments / pricing
- GD or WAT simulation
- Multi-language support
- Analytics / admin panel

## Architecture Principle

Engine and presentation layer are architecturally separated from day
one, even though only the text interface is built in v0.1. The engine
should not assume a specific delivery mechanism — this keeps voice/
avatar additive later rather than requiring a rebuild. See Decision
#002 for full reasoning and the Stage 1 / Stage 2 validation plan.

## Success Criteria for This Stage

Not pricing, not user count. Success is qualitative and specific:
students describing the *questioning itself* as sharp, adaptive, or
challenging — not merely "the app was nice to use." Per the design
principle adopted in Decision #002: presentation polish does not count
as evidence the engine works.

## Explicit Non-Goal

This is not a race to feature completeness. If Stage 1 shows the engine
is weak — generic questions, follow-ups that ignore what was actually
said (the single most common complaint found in competitor research) —
that is a valid and important outcome. Fix the engine before adding any
interface richness. Do not let a disappointing Stage 1 result trigger
an instinct to "add an avatar to make it feel more real" — that would
repeat the exact mistake this decision was designed to avoid.
