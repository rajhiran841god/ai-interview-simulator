# Milestone 2 — Interview Intelligence Engine Architecture

**Status:** Draft — pending Architecture Review Gate
**Design only. No implementation code.**

---

## 1. Core Philosophy

The engine does not optimize for asking questions. It optimizes for
collecting sufficient evidence to make a transparent, competency-based
evaluation of a candidate. Its governing question, repeated every turn:

> "What is my current confidence in evaluating each competency, and
> what evidence do I still need to reach a confident, evidence-backed
> assessment?"

## 2. Module Breakdown

### 2.1 Resume Understanding
- **Purpose:** Extract structured candidate background from an uploaded resume.
- **Input:** Raw resume file (PDF/text).
- **Output:** Structured profile — roles, projects, claimed skills, quantifiable achievements, gaps/ambiguities worth probing.
- **Responsibilities:** Parsing, normalization, flagging vague or unverifiable claims (e.g. "led a team" with no team size or outcome) as candidate probe targets.
- **Interacts with:** JD Understanding (to compute fit), Competency Model (seeds initial evidence targets), Question Generator.

### 2.2 JD Understanding
- **Purpose:** Extract the target role's actual requirements.
- **Input:** Job description text.
- **Output:** Required competencies, seniority signals, role-specific emphasis (e.g. a consulting JD emphasizes structured problem solving over technical depth).
- **Responsibilities:** Determines which competency set applies for this interview (not hardcoded — derived per JD, per the Milestone 2 brief).
- **Interacts with:** Competency Model (defines which competencies matter and their relative weight for this session).

### 2.3 Competency Model / Confidence Tracker
- **Purpose:** Maintain the live state of "how confident are we in evaluating each competency" for the current candidate.
- **Input:** Competency list (from JD Understanding), evidence updates (from Evaluation Engine).
- **Output:** Per competency: `evidence_count`, `positive_evidence` (refs into Evidence Graph), `contradictory_evidence` (refs into Evidence Graph), and the derived `confidence` score (internal signal — see Section 5, never shown raw to the student).
- **Responsibilities:** Score initialization, update rules, exposing "lowest-confidence competency" to the Reasoning Engine each turn. Deliberately kept to these four fields for the pilot — things like "open questions remaining" or "last updated" are computed on demand from the Evidence Graph when needed rather than stored as separate state, to avoid redundant fields that can drift out of sync during a fast build.
- **Interacts with:** Evidence Graph, Reasoning Engine.

### 2.4 Evidence Graph
- **Purpose:** Structured record linking every piece of evidence to its source.
- **Input:** Extracted evidence from each candidate answer.
- **Output:** A queryable graph/table: `evidence -> competency -> source question -> source answer excerpt -> supports/contradicts`.
- **Responsibilities:** This is what makes Evidence-Based Feedback (Gate #4) structural rather than reconstructed after the fact. Every feedback statement is generated by querying this graph, not by asking the model to "summarize how it went." Also flags when new evidence **contradicts** an earlier answer for the same competency (e.g. "I led the project" vs. later "my manager made all decisions") — kept as a simple boolean/reference flag on the evidence entry for the pilot, not a full contradiction-resolution algorithm. This flag is what gives the Reasoning Engine grounds to choose "challenge inconsistency" as a strategy.
- **Interacts with:** Evaluation Engine (writes to it, including conflict detection), Feedback Generator (reads from it), Reasoning Engine (reads conflict flags).

### 2.5 Reasoning / Decision Engine
- **Purpose:** The core adaptive logic. Each turn, decides what to do next.
- **Input:** Current confidence scores, evidence graph state (including any conflicting evidence flags), conversation memory.
- **Output:** A structured decision object: target competency, **decision strategy** (probe deeper / challenge inconsistency / verify / switch competency / wrap up competency), and a human-readable reason. Not just a question — strategy and reason stay separate from Question Generator's phrasing job.
- **Responsibilities:** Implements the two-layer state machine's dynamic loop (Section 3). Evaluates the stopping condition (Section 4) each turn. When the Evidence Graph shows conflicting evidence for a competency, "challenge inconsistency" becomes an available strategy, not just "probe deeper."
- **Interacts with:** Competency Model (reads), Question Generator (writes target + strategy + reason), Logging (writes the full reasoning trace).
- **Note:** An earlier review suggested a separate "Interview Strategy Layer" module between Reasoning and Question Generator. Adopting the underlying idea (decision separated from language) via the `decision_strategy` field above rather than a new module — same benefit, less module-count overhead for a pilot build. Revisit as a true separate module only if Milestone 3 shows the strategy logic outgrowing this.

### 2.6 Question Generator / Cross-Questioning Engine
- **Purpose:** Turn a target competency + evidence gap into an actual question.
- **Input:** Target competency, specific missing evidence, conversation memory (to avoid repetition, to reference prior answers for cross-questioning).
- **Output:** Natural-language question, tagged with its target competency and reasoning ID.
- **Responsibilities:** Distinguishes a fresh question (new competency) from a cross-question (probing deeper into an already-raised but under-evidenced answer — e.g. "You mentioned leading the team — what specifically did you do when there was disagreement?").
- **Interacts with:** Reasoning Engine (receives target), Conversation Memory (reads/writes).

### 2.7 Conversation Memory
- **Purpose:** Full turn-by-turn record of the interview so far.
- **Input/Output:** Question/answer pairs, in order.
- **Responsibilities:** Available to every module that needs context — prevents repeated questions, enables cross-questioning, feeds the transcript.
- **Interacts with:** Nearly every module reads this.

### 2.8 Evaluation Engine
- **Purpose:** Given a candidate's answer, extract evidence and update confidence.
- **Input:** Candidate's raw answer + the question's target competency/reasoning context.
- **Output:** Extracted evidence (written to Evidence Graph), updated confidence score (written to Competency Model).
- **Responsibilities:** This is where "does the answer actually address the gap" gets judged — must be able to recognize a non-answer, a deflection, or a strong answer, not just log text.
- **Interacts with:** Evidence Graph, Competency Model, Logging.

### 2.9 Feedback Generator
- **Purpose:** Produce the final, student-facing report.
- **Input:** Full Evidence Graph, final confidence state.
- **Output:** Per-competency qualitative feedback, each claim traceable to a specific question/answer (Gate #4).
- **Responsibilities:** Never emits raw confidence numbers to the student (Section 5). Every statement must cite its evidence source.
- **Interacts with:** Evidence Graph (primary source), Conversation Memory (for quoting/paraphrasing specific moments).

### 2.10 Logging / Trace Recorder
- **Purpose:** Structural, queryable record of every reasoning step (Gate #5).
- **Input:** Every Reasoning Engine decision, every Evaluation Engine update.
- **Output:** See schema in Section 6.
- **Responsibilities:** Not an afterthought — every other module writes to this as a side effect of doing its job, not as a separate manual step.

## 3. State Machine

### Layer 1 — Fixed Lifecycle (linear, does not change)
```
START → Consent → Resume Upload → JD Upload → Greeting → Interview
      → Evaluation → Feedback → END
```

### Layer 2 — Dynamic Reasoning Loop (inside "Interview")
```
Evidence Graph + Confidence state
        │
        ▼
Identify lowest-confidence competency with insufficient evidence
        │
        ▼
Reasoning Engine selects target + generates "reason"
        │
        ▼
Question Generator produces question or cross-question
        │
        ▼
Candidate answers
        │
        ▼
Evaluation Engine extracts evidence, updates confidence
        │
        ▼
Check stopping condition (Section 4)
        │
   ┌────┴────┐
   ▼         ▼
 Continue   Exit → Evaluation → Feedback
```
No hardcoded competency order. The engine recomputes the target each
turn based on current state, not a script.

## 4. Stopping Condition

- **Minimum question floor:** 6 questions, regardless of how quickly
  confidence rises. Rationale: an interview that ends in 2-3 questions
  will feel dismissive and undermine trust in the pilot, even if the
  candidate answered strongly.
- **Target threshold:** average confidence across all tracked
  competencies ≥ 85%, AND no single competency below 60% (prevents a
  high average from masking one totally unprobed competency).
- **Maximum question ceiling:** 18 questions. If threshold isn't met by
  then, exit anyway. A competency that remains low-confidence after 18
  questions is itself a valid, evidence-backed finding ("insufficient
  evidence was gathered on X despite repeated probing" is a legitimate
  and honest thing to report) — not a failure requiring more questions.
- These bounds (6/18, 85%/60%) are **pilot defaults, implemented as
  configurable values** (e.g. a config/settings entry), not hardcoded
  constants — expect to tune them after Stage 1 pilot feedback (per
  Decision #002), without needing a code change to do so.

## 5. Confidence Score Constraint (Critical)

Internal competency confidence scores are engine-internal reasoning
signals representing **"confidence that sufficient evidence has been
collected to assess this competency"** — not an objective measure of
the candidate's true ability.

**Hard rules, enforced structurally:**
- Confidence scores/percentages are never serialized into any
  student-facing API response or UI.
- The Feedback Generator's only permitted input from the Competency
  Model is *which competencies are considered sufficiently evidenced*
  (a boolean/categorical signal) — not the raw score — forcing the
  qualitative, evidence-grounded write-up to be the only output path.
- This distinction is documented as ADR-0006 specifically so it can't
  be quietly reversed later by someone optimizing the feedback UI.

## 6. Logging / Reasoning Trace Schema

Every question produces a record:

| Field | Description |
|---|---|
| interview_id | |
| question_id | |
| target_competency | |
| decision_strategy | probe deeper / challenge inconsistency / verify / switch competency / wrap up competency |
| confidence_pre | Confidence before this question |
| evidence_missing | What gap this question targets |
| reason_for_asking | Human-readable justification |
| prompt_version | |
| model_version | |
| candidate_answer | |
| evidence_extracted | What the Evaluation Engine found |
| confidence_post | Confidence after this question |

This makes "why did the engine ask that" answerable by querying a
record, not by guessing.

## 7. Pilot Learning Loop

```
Interview Complete → Evaluation Report generated → Pilot Feedback Form
  (Did this feel like a real interview? Which question felt most/least
  realistic? Was any feedback especially useful? What frustrated you?
  Would you practice again?)
        │
        ▼
Stored with a direct link back to the interview_id and full trace,
surfaced in a simple founder-reviewable view (not raw log scraping).
```

## 8. Architecture Decision Records

**ADR-0001 — Engine is core IP.** The Interview Intelligence Engine,
not the interface, is the defensible product. Presentation layers are
replaceable; the engine is not.

**ADR-0002 — Provider abstraction.** No module calls an LLM SDK
directly; all calls route through a single abstraction so providers
can be swapped without touching business logic.

**ADR-0003 — Engine/presentation separation.** Text, voice, and avatar
are interchangeable delivery layers on top of one engine. v0.1 ships
text only.

**ADR-0004 — Build-first pilot strategy.** Given the strategic pivot
documented in Decision Log #002, v0.1 is validated via a real pilot
with 20 students rather than exclusively via pre-build interviews.

**ADR-0005 — Evidence-driven reasoning.** The engine selects questions
by identifying the lowest-confidence, least-evidenced competency each
turn, rather than following a fixed question script.

**ADR-0006 — Confidence scores are internal reasoning signals, not
ability measures.** See Section 5. Never shown raw to students; always
translated into evidence-grounded qualitative feedback.

## 10. Enforceable Rule — Provider Independence

**No module except a single `ProviderAdapter` may import the Anthropic
SDK (or any future LLM provider's SDK) directly.** This is a code
review rule for Milestone 3, not just a design intention — it's what
makes Gate #3 actually true rather than aspirational.

## 11. Implementation Order (Milestone 3)

Corrected from a prior draft that suggested starting with the
Reasoning Engine. Building in dependency order, with observability
first, reduces the risk of debugging everything at once:

1. Resume Understanding — structured extraction
2. JD Understanding — competency extraction
3. Conversation Memory — interview context
4. Evidence Graph — data model and storage (including conflict flag)
5. Logging / Trace Recorder — built early so every later module is observable from day one
6. Evaluation Engine — evidence extraction and conflict detection
7. Competency Model / Confidence Tracker
8. Reasoning / Decision Engine (including decision_strategy)
9. Question Generator / Cross-Questioning Engine
10. Feedback Generator

Each module should be independently testable before the next one is
built on top of it.
